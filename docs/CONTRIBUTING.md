# CONTRIBUTING.md — 협업 규칙 (초안)

> **이 문서는 초안입니다.** 2026-05-27~28 밤샘 회의 및 이후 팀 논의를 거쳐 확정됩니다.
> 현재 내용은 작업을 시작할 수 있는 최소 합의안이며, 회의 결과로 자유롭게 수정됩니다.

## 브랜치 전략

| 브랜치 | 역할 |
| --- | --- |
| `main` | 배포 가능한 안정 버전. **직접 푸시 금지**, PR로만 머지. |
| `develop` | 개발 통합 브랜치. 평소 작업은 여기로 머지된다. |
| 작업 브랜치 | 아래 네이밍 규칙으로 짧게 유지. |

### 작업 브랜치 네이밍

- `feat/기능명` — 새로운 기능
- `fix/버그명` — 버그 수정
- `docs/문서명` — 문서 추가·수정
- `chore/항목명` — 설정·빌드·잡일
- `refactor/대상명` — 동작 변경 없는 리팩터링
- `test/대상명` — 테스트 추가·수정

예) `feat/portfolio-simulator`, `fix/ips-input-validation`, `docs/contributing-draft`

## 작업 흐름

1. `develop` 브랜치를 최신 상태로 받는다. (`git pull origin develop`)
2. `develop`에서 작업 브랜치를 딴다. (`git checkout -b feat/기능명`)
3. 작업 → 커밋(컨벤션 준수) → 푸시.
4. GitHub에서 PR 생성 (`develop` ← `feat/기능명`). PR 템플릿을 채운다.
5. **최소 1명 리뷰 승인 후** `develop`에 머지.
6. 배포 시점이 되면 `develop` → `main` PR을 별도로 만든다.

## 커밋 메시지 컨벤션

`타입: 설명` 형식, **한국어**로 작성.

| 타입 | 용도 |
| --- | --- |
| `feat` | 새로운 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `chore` | 설정·빌드·잡일 (코드 동작 변화 없음) |
| `refactor` | 동작 변경 없는 코드 정리 |
| `test` | 테스트 추가·수정 |

예시
- `feat: 포트폴리오 시뮬레이션 입력 폼 추가`
- `fix: IPS 입력 값 유효성 검사 오류 수정`
- `docs: CONTRIBUTING 초안 작성`

## PR 규칙

- **PR 템플릿을 빠짐없이 채운다.** (`.github/pull_request_template.md`)
- **너무 큰 PR 지양.** 가능하면 기능 단위로 쪼개서 리뷰가 가능한 크기로.
- **셀프 머지 지양.** 자신의 PR을 자기가 머지하지 않는다 (긴급 핫픽스 등 예외 상황 제외).
- 리뷰 코멘트는 가급적 24시간 내 회신.
- PR 머지 후 작업 브랜치는 삭제한다.
